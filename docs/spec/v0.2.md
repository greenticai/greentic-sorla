# SoRLa v0.2 Specification Notes

## Purpose

SoRLa v0.2 extends the authoring language so packages can describe event-native
models, projection-based read models, provider requirements, and records that
blend native workflow state with authoritative external systems of record.

The v0.2 model stays additive relative to the familiar package shape:

- `records`
- `events`
- `actions`
- `policies`
- `approvals`
- `views`
- `flows`
- `migrations`

## New Semantics

### `source`

Records now declare `source: native | external | hybrid`.

- `native`
  SoRLa-managed record is authoritative for writable state.
- `external`
  An external system is authoritative. SoRLa may cache, reference, enrich, and
  orchestrate around the record, but does not claim canonical ownership.
- `hybrid`
  Authority is field-level. External systems remain authoritative for declared
  external fields, while SoRLa remains authoritative for local overlays,
  workflow state, approvals, derived state, and precedents.

For additive v0.1 compatibility, omitted `source` defaults to `native` and
produces a warning rather than requiring a separate legacy parser mode.

### `external_ref`

External and hybrid records declare `external_ref` so the package can identify
the external system and key used to resolve the authoritative record.

Example:

```yaml
external_ref:
  system: crm
  key: tenant_id
  authoritative: true
```

### `authoritative`

`external_ref.authoritative: true` marks that the referenced external system is
the source of truth for externally owned fields. This is required for `hybrid`
records in the current parser/validator.

### Field-level authority

Hybrid records must assign `authority: local | external` to each field. This
avoids record-level ambiguity and keeps conflict handling explicit.

## Events and Projections

Events are first-class language objects and projections are explicit read models
derived from events rather than implicit table snapshots.

Example fit-out case event:

```yaml
events:
  - name: FitOutCaseOpened
    record: FitOutCase
    kind: domain
    emits:
      - name: case_id
        type: string
      - name: opened_at
        type: timestamp
```

Projection example:

```yaml
projections:
  - name: FitOutCaseCurrentState
    record: FitOutCase
    source_event: FitOutCaseOpened
    mode: current-state
```

## Provider Requirements

Provider requirements stay abstract in `greentic-sorla`. Packages declare
required capability categories without binding concrete providers:

```yaml
provider_requirements:
  - category: storage
    capabilities:
      - event-log
      - projections
```

Concrete provider packs and bindings belong in `greentic-sorla-providers`.

## Migrations and Compatibility

`migrations` in v0.2 are not framed primarily as SQL-style schema rewrites.
They describe compatibility intent and projection evolution for Git-driven
rollout.

Example:

```yaml
migrations:
  - name: tenant-projection-v2
    compatibility: additive
    projection_updates:
      - TenantCurrentState
```

Compatibility is intended to be explicit and reviewable. Consumers should reason
about additive, backward-compatible, and breaking changes rather than assuming
table rewrite semantics.

## Examples

### Native record

```yaml
records:
  - name: ApprovalPolicy
    source: native
    fields:
      - name: policy_id
        type: string
```

### External record

```yaml
records:
  - name: LeaseCase
    source: external
    external_ref:
      system: sharepoint
      key: lease_case_id
      authoritative: true
    fields:
      - name: lease_case_id
        type: string
```

### Hybrid record

```yaml
records:
  - name: Tenant
    source: hybrid
    external_ref:
      system: crm
      key: tenant_id
      authoritative: true
    fields:
      - name: tenant_id
        type: string
        authority: external
      - name: approval_state
        type: string
        authority: local
```
