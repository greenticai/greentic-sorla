# PR 02: Add SoRLa v0.2 language semantics for event-native, external-aware models

    **Repository:** `greenticai/greentic-sorla`

    ## Objective

    Implement the SoRLa v0.2 language update so the language itself can express:
- event-first storage semantics
- projection-based read models
- authoritative external references
- hybrid native/external records
- provider requirements
- compatibility and evolution metadata suitable for Git-driven rollout

    ## Why this PR exists

    Without these language additions, SoRLa remains implicitly schema-oriented and cannot cleanly describe the FoundationDB/event-native model or the external system-of-record model required for KAFD. The language must carry the semantics rather than leaving them to informal convention.

    ## Scope

    Extend the authoring model and internal representation to support new concepts while preserving the spirit of the v0.1 package layout. Keep the language readable and YAML-authored. New or revised concepts should include:
- `source: native | external | hybrid`
- `external_ref`
- `authoritative`
- `events`
- `projections`
- updated `migrations` semantics oriented around compatibility/projection evolution
- provider requirement metadata
Make sure old v0.1-style packages can still be parsed or at least produce clear upgrade guidance.

    ## Deliverables

    - language specification notes under `docs/spec/v0.2.md`
- parser-facing schema definitions / AST nodes for:
  - `source`
  - `external_ref`
  - `authoritative`
  - `events`
  - `projections`
  - `provider_requirements`
- examples of native, external, and hybrid records
- examples of event declarations and projection declarations
- compatibility notes for v0.1 -> v0.2

    ## Implementation notes for Codex

    Prefer additive evolution over breaking redesign. Preserve the familiar top-level SoRLa package structure so existing mental models still fit:
- `records`
- `events`
- `actions`
- `policies`
- `approvals`
- `views`
- `flows`
- `migrations`
Then extend with the new semantics rather than inventing an entirely new language. Include example snippets for:
- a hybrid `Tenant`
- a fit-out case with emitted events
- a projection representing current case state
Document how `external_ref` works conceptually: SoRLa may enrich and orchestrate, but the external system remains authoritative when marked as such.

    ## Acceptance criteria

    - Spec doc clearly explains every new v0.2 concept
- Example YAML snippets parse into the new AST structures
- Tests cover at least one native record, one external record, and one hybrid record
- Event and projection declarations are represented in the language model
- Migration semantics docs no longer imply SQL-style schema rewrites as the primary model

    ## Non-goals

    - Full compiler output
- Wizard generation
- Runtime/provider implementation
- KAFD-specific package content

    ## Suggested files / areas to touch
