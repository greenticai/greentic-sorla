# PR 16 — SoRLa Designer Extension contributes generated Designer node types

## Repository

`greenticai/greentic-sorla`

## Objective

Extend the SoRLa Designer Extension so it contributes Designer node types generated from SoRLa agent endpoints.

This allows Designer to add "Record monthly rent payment" or "Submit claim" as normal nodes while runtime execution goes through locked SoRLa endpoint metadata and the configured downstream component operation.

This PR depends on PR-15. The current repository has `crates/greentic-sorla-designer-extension` as a deterministic JSON adapter and does not vendor a `greentic-designer-sdk` WIT package. Keep this PR on the existing adapter boundary until the real SDK/WIT is available.

## Requirements

The extension should provide node type contributions using a generic contribution shape:

```json
{
  "nodeTypes": []
}
```

Do not add SoRLa-specific SDK fields to a shared SDK schema. SoRLa-specific runtime binding data belongs inside node type metadata/config fields generated by PR-15.

## Possible implementation modes

Support one or both:

### Static pack-bundled node types

The extension includes or references:

```text
assets/sorla/designer-node-types.json
```

### Dynamic tool-generated node types

The extension exposes a tool:

```text
list_designer_node_types
```

that returns node types for a generated/loaded SoRLa model using the public facade crate.

## Tool: `list_designer_node_types`

Input:

```json
{
  "model": {},
  "operation": "invoke_locked_action",
  "component_ref": "oci://ghcr.io/greenticai/components/component-sorx-business:0.1.0"
}
```

Output:

```json
{
  "nodeTypes": [],
  "diagnostics": []
}
```

## Tool: `generate_flow_node_from_node_type`

Input:

```json
{
  "node_type_id": "sorla.agent-endpoint.record_rent_payment",
  "step_id": "record_payment",
  "value_mappings": {
    "amount": "$.state.amount"
  }
}
```

Output:

A generic Designer/Greentic flow node JSON object that calls the configured component with locked endpoint metadata. If a first-class Designer flow schema fixture exists in this repo by implementation time, validate against it; otherwise keep the adapter output explicitly documented as generic JSON.

## Validation

The extension should validate:

1. node type ID exists
2. required endpoint input fields are mapped or have defaults
3. endpoint ref includes id/version/package/hash
4. generated flow node uses the configured component ref and operation
5. no runtime free-text intent selection

## Tests

Add tests for:

- list node types from the generated landlord/tenant SoRLa model
- generated node type validates against the PR-15 schema or SDK fixture/schema if available
- generate flow node includes endpoint ID/version/package/hash
- missing mapping returns diagnostics
- no free-text runtime action field
- adapter manifest exposes the new tools deterministically
- wasm build remains compatible with `wasm32-wasip2`
